# Weekly Meal Plan Prompt

You are generating a weekly dinner plan for the household configured in this repo. This runs every Saturday morning via GitHub Actions. Follow these steps in order. Do not skip steps.

## Step 1: Read state

Read these files from the repo root:

- `config.yml` — recipients, timezone
- `family-context.md` — standing household preferences (ages, dislikes, kid blockers, equipment, etc.)
- `meal-history.md` — running log of past meals
- `current-week.md` — last week's plan (if it exists)
- `notes.md` — forward-looking notes from the household about the upcoming week (head-count changes, travel, school events, ingredients to use up, etc.)
- `requests-inbox.md` — captured replies to the Friday request email (optional; only present if the inbox-fetch step found something this week)

## Step 2: Determine the target week

The target week is the upcoming Monday through the following Sunday.

- Today is Saturday (the GitHub Action is scheduled for Saturdays). The target week starts 2 days from now.
- Compute the actual date of each day (Mon/Tue/Wed/Thu/Fri/Sat/Sun) using the timezone in `config.yml`.
- Use real YYYY-MM-DD dates in all output.

## Step 3: Parse history

From `meal-history.md`, extract:

1. **All meals in the last 30 days** — ineligible for this week (the no-repeat rule).
2. **Meals rated `hit` that appear 2+ times across all history** — these are mainstay favorites. Up to 1 may appear this week, even if cooked recently. If you use this exception, call it out explicitly in the plan notes.
3. **Meals rated `miss`** — do not suggest again unless substantially reworking. If reworking, say so.
4. **Notes on recent meals** — if a note says "kids refused broccoli" or "too spicy," adjust accordingly.

If `meal-history.md` has no real entries (just the example block), treat history as empty.

## Step 4: Family context

Use what you read from `family-context.md` in Step 1 — household composition, dietary constraints, adult dislikes, kid blockers, kid mainstays, kitchen equipment, and skill level all come from there. Treat every bullet in that file as a hard constraint unless it's clearly framed as a preference rather than a rule.

## Step 5: Generate 7 dinners

Constraints:

1. **No repeats within the last 30 days**, with at most 1 mainstay-favorite exception.
2. **Active time:** 30 min ideal, 60 min acceptable. Passive time (braise, slow cooker, smoker) unlimited — flag long-passive meals.
3. **Protein spread:** aim for chicken x2, beef x1, pork x1, fish/seafood x1 (fish sticks or similarly kid-friendly prep is OK), pasta/vegetarian x1, flex x1. Adjust based on what's been heavy in recent history.
4. **Shared ingredients:** prefer meals that share fresh ingredients to tighten the grocery list. Call out overlaps.
5. **Audible-friendly sequencing:** perishable meals (fresh fish, delicate produce) early; shelf-stable or freezer-friendly proteins later. If Wednesday becomes takeout, nothing should rot.
6. Format repetition (pasta night every week) is OK; specific dishes should rotate.
7. **Apply `notes.md`.** For each note in the "Active notes" section, work out which day(s) of the target week it applies to and adjust accordingly — head-count changes (smaller portions, easier prep), known absences (kid-friendly version not needed that night), school/work events (lighter meal, or skip entirely), ingredients to use up (lean toward dishes that include them). If a note is ambiguous about dates, use your best judgment; if it's clearly outside the target week, ignore it for now (Step 9 will handle pruning). Every applied note must be called out in the "Plan notes" section of `current-week.md` so it's visible.
8. **Apply `requests-inbox.md` if present.** This file is generated by `scripts/fetch_requests.py` from replies to the Friday request email. Treat the three sections with different weights:
   - **Must have / Must avoid** — hard constraints. Honor every bullet. If a "must have" conflicts with another rule (e.g. the 30-day no-repeat or a `family-context.md` constraint), prefer the must-have and note the override in Plan notes. If a set of constraints is collectively impossible (e.g. "tacos and no beef and no chicken"), pick the closest legal interpretation and explain the deviation in Plan notes.
   - **Soft preferences** — strong but not absolute. Bias the week toward these; deviate only if needed to satisfy a hard rule or `family-context.md` constraint.
   - **Use up** — bias toward dishes that incorporate the listed ingredients, ideally early in the week so they don't spoil.
   - Every applied or overridden request must be enumerated in the "Plan notes" section of `current-week.md`. If `requests-inbox.md` doesn't exist, skip this step silently.

## Step 6: Pick recipe links

For each of the 7 meals, find a recipe URL matching the dish. Prefer: Serious Eats, NYT Cooking, Bon Appétit, America's Test Kitchen, Smitten Kitchen, Kenji López-Alt, Ina Garten, established food blogs.

**Do not WebFetch URLs to verify them.** A workflow step downstream (`scripts/validate_recipe_urls.py`) checks every URL with HTTP requests after you finish — that's its job, not yours. WebFetch is unreliable in this environment and will burn through your turn budget; in 2026-04-25's run, retrying WebFetch on transient errors caused max-turns exhaustion before the plan was written.

How to pick URLs:

1. Use your own knowledge of well-known recipes from the preferred sites — many of them you already know the canonical URL for (e.g. Smitten Kitchen, Cookie and Kate, Half Baked Harvest follow stable URL patterns).
2. If you're unsure, do **one** WebSearch per dish to surface a candidate. Trust the top result from a reputable source listed above.
3. If no good candidate appears, omit the URL and include a detailed one-line description in `**Recipe:**` instead (e.g. "no link — pan-sear chops, deglaze with apple cider + Dijon, finish with thyme butter").

Budget: aim to spend no more than ~10 turns total on URL selection. The downstream validator will catch any 404s and bot-blocks for you.

## Step 7: Write `current-week.md`

Overwrite `current-week.md` with **exactly** this structure. The format matters because `scripts/send_daily_reminder.py` parses it.

```markdown
# Current Week: YYYY-MM-DD to YYYY-MM-DD

## Monday YYYY-MM-DD — [Dish Name]
- **Description:** [one line]
- **Active time:** [X min]  |  **Total time:** [X min, or "flag: long passive"]
- **Protein:** [type]
- **Recipe:** [URL or "no link — see description"]
- **Notes:** [kid modifications, ingredient overlaps, flexibility notes]

## Tuesday YYYY-MM-DD — [Dish Name]
[...same structure...]

[...through Sunday...]

## Ingredient overlaps
- [e.g., "parsley used Mon + Thu"]

## Pantry check — make sure you have
- [items the plan assumes are in the pantry]

## Shopping list

### Produce
- [items with quantities]

### Meat & Seafood
- [items]

### Dairy & Eggs
- [items]

### Bakery
- [items]

### Pantry / Dry Goods (new purchases only)
- [items]

### Frozen
- [items]

## Plan notes
- [mainstay-exception used? sequencing notes? anything unusual?]
```

**Critical format requirements for the daily reminder parser:**

- Each day heading MUST follow the pattern: `## [Weekday] YYYY-MM-DD — [Dish Name]` with an em-dash (—) between the date and dish.
- The weekday names must be full English: Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, Sunday.
- The field labels MUST be exactly: `**Description:**`, `**Active time:**`, `**Protein:**`, `**Recipe:**`, `**Notes:**`.

If you deviate from this format, the daily reminder script will fail silently.

## Step 8: Update `meal-history.md`

Append a new entry for each of the 7 meals, using the correct date per day. Each entry mirrors the per-day block from `current-week.md` plus the running `Rating` / `Notes` fields the user fills in during/after the week:

```markdown
### YYYY-MM-DD

- **Meal:** [Dish Name]
- **Description:** [same one-liner from current-week.md]
- **Active time:** [X min]  |  **Total time:** [X min, or "flag: long passive"]
- **Protein:** [type]
- **Recipe:** [URL or "no link — see description"]
- **Rating:**
- **Notes:**
```

`Rating` and `Notes` stay blank — they get filled in during/after the week. The other fields must match what you wrote in `current-week.md` for the same day verbatim, since `send_daily_reminder.py` falls back to history when `current-week.md` doesn't cover today (e.g. on Sunday, after Saturday's run already overwrote the prior week's plan).

If `meal-history.md` only has the example entries, leave those alone (they have dates that will age out of the 30-day window naturally). Just append the new 7.

Do not overwrite existing entries. Append only.

## Step 9: Prune `notes.md`

Open `notes.md`. For each bullet in the "Active notes" section:

- If the note references a date or date range that falls **within or before** the target week you just planned, delete that bullet.
- If the note references a date **after** the target week, leave it alone (it's for a future plan).
- If the note has no date and was a one-shot you applied this week (e.g. "use up the arborio rice"), delete it.
- If the note has no date and is a standing preference for several weeks (e.g. "someone avoiding dairy through May"), leave it alone.

Rewrite `notes.md` with the kept bullets only. Preserve the file's header, the "How to write a note" section, the `## Active notes` heading, and the HTML comment marker — only edit the bullet list under it. If no bullets remain, leave the section empty (just the heading + comment marker).

If `notes.md` had no Active notes to begin with, skip this step entirely — don't touch the file.

After pruning `notes.md`, delete `requests-inbox.md` if it exists. The requests it carried are one-shot inputs consumed by this week's plan; they must not leak into next week. Do not commit a deletion (the workflow's commit step only adds the plan files); the file is gitignored anyway.

## Step 10: Finish

Make sure `current-week.md`, `meal-history.md`, and (if you edited it) `notes.md` are written to disk. Do **not** commit — the GitHub Action will commit and push them after you exit. Do not attempt to run `git` commands; they aren't available to you in this run.
